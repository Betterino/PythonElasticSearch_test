# Document Search Service

Простой поисковый сервис по текстам документов. Данные хранятся в PostgreSQL,
полнотекстовый поиск — через Elasticsearch.

## Стек

- **FastAPI** — веб-фреймворк (async, автогенерация OpenAPI-схемы)
- **PostgreSQL + SQLAlchemy 2.0 (async, asyncpg)** — хранилище документов, источник правды
- **Elasticsearch 8.15** — поисковый индекс по тексту документов
- **pytest + pytest-asyncio + httpx** — функциональные тесты
- **Docker Compose** — оркестрация всего стека

## Запуск

### 1. Поднять стек

```bash
docker compose up --build -d
```

Поднимутся три сервиса: `postgres`, `elasticsearch`, `app`. Дождитесь, пока все
будут в статусе `healthy`:

```bash
docker compose ps
```

Таблицы в Postgres и индекс в Elasticsearch создаются автоматически при старте
приложения (см. `lifespan` в `app/main.py`).

### 2. Загрузить данные из CSV

Положите файл с данными в `data/posts.csv`, затем:

```bash
docker compose exec app python -m scripts.import_csv data/posts.csv
```

Скрипт читает CSV, парсит поле `rubrics` (строка вида
`"['VK-123', 'VK-456']"` — сохранённая репрезентация Python-списка, парсится
через `ast.literal_eval`), пишет документы в Postgres батчами по 200 и
одновременно индексирует их в Elasticsearch через `bulk`-запрос.

### 3. Проверить, что всё работает

Swagger UI: [http://localhost:8000/docs](http://localhost:8000/docs)

Или руками:

```bash
curl "http://localhost:8000/search?q=кот"
curl -X DELETE "http://localhost:8000/documents/1"
```

## API

Полная OpenAPI-схема — в [`docs.json`](./docs.json) или на `/docs` у
запущенного сервиса.

### `GET /search?q=<текст>`

Ищет документы по тексту, возвращает не более 20 штук, отсортированных по
`created_date` (сначала новые). В ответе — все поля документа из БД (`id`,
`rubrics`, `text`, `created_date`).

### `DELETE /documents/{id}`

Удаляет документ из БД и из поискового индекса. `404`, если документа с таким
id нет.

## Тесты

Тесты бьют по реальным Postgres/Elasticsearch (в отдельных, тестовых
БД/индексе) — не по мокам. Поэтому перед запуском стек должен быть поднят.

Тестовая база данных `docsearch_test` создаётся автоматически при **первом**
поднятии стека (init-скрипт `docker/init-test-db.sql`, подключённый как volume
к контейнеру `postgres` — Postgres выполняет такие скрипты один раз, при
инициализации пустого volume). Если стек уже поднимался раньше на этой
машине и volume не пустой, создайте базу вручную:

```bash
docker compose exec postgres psql -U docsearch -d docsearch -c "CREATE DATABASE docsearch_test;"
```

Запуск тестов:

```bash
docker compose exec app pytest tests/ -v
```

Что покрыто:

- юнит-тесты чистой функции `parse_rubrics` (без БД/ES) — нормальный и
  некорректный вход;
- поиск находит документ по совпадению в тексте;
- результаты отсортированы по `created_date` по убыванию;
- в ответе присутствуют все поля документа, а не только текст;
- удаление убирает документ и из БД, и из поисковой выдачи;
- HTTP-слой: `200` на успешный поиск, `404` на удаление несуществующего id.

## Архитектурные решения и известные ограничения

### Поле `rubrics` не участвует в поисковом индексе

По структуре из задания индекс Elasticsearch содержит только `id` и `text` —
`rubrics` там нет. Реализовано строго по спецификации: `rubrics` хранится
только в БД и возвращается в ответе `/search` как обычное поле документа, но
не участвует в поиске и не фильтрует выдачу. Если нужен поиск/фильтрация по
рубрикам — это несложно добавить как `keyword`-поле в mapping индекса (в
отличие от `text`, `keyword`-поля не токенизируются и ищутся точным
совпадением) — писать не стал, так как явного требования в задании нет.

### Сортировка по дате при поиске через широкий пул кандидатов

В индексе ES нет поля `created_date` (по спецификации индекса), а
сортировать результат нужно именно по нему. Elasticsearch по умолчанию
упорядочивает результаты по релевантности (score), а не по дате — прямая
пересортировка top-20 по дате могла бы вытеснить релевантный, но со старым
score документ.

Решение: из ES запрашивается не 20, а 200 кандидатов
(`CANDIDATE_POOL` в `search_service.py`) по релевантности, затем среди них
делается `SELECT ... ORDER BY created_date DESC LIMIT 20` уже в Postgres.
Это компромисс, а не идеальное решение — теоретически релевантный документ
за пределами топ-200 по score всё ещё может быть пропущен. Более строгий
вариант — индексировать `created_date` и в ES, сортировать средствами
самого Elasticsearch, — но это отступление от заданной структуры индекса.

### Postgres как источник правды при рассинхроне с ES

`search_documents()` получает id кандидатов из ES, но финальные данные и
итоговую фильтрацию делает через Postgres (`WHERE id IN (...)`). Если
документ есть в ES, но отсутствует в БД (например, из-за сбоя при
удалении), он просто не попадёт в выдачу — не вызовет ошибку.

При удалении (`delete_document`) сначала удаляется запись в Postgres, затем
— в Elasticsearch. Если удаление из ES не удалось — ошибка логируется, но
не приводит к откату: raз Postgres — источник правды, а поиск всё равно
дофильтровывает по БД, "осиротевший" документ в ES не более чем безвредный
мусор, который со временем стоит подчистить.

### Импорт CSV не транзакционен между батчами

Импорт пишет документы в Postgres и индексирует их в ES батчами по 200.
Если сбой (например, сетевая ошибка при обращении к ES) происходит
в середине импорта — уже обработанные батчи остаются в БД и в индексе,
необработанные — нет. Скрипт можно запустить повторно, но это создаст
дубликаты (сейчас нет проверки на уже импортированные записи). Для объёма
данных в тестовом задании (1500 строк) это не является проблемой на
практике, но при реальном использовании стоит добавить идемпотентность
(например, по внешнему ключу документа) или retry с точкой возобновления.

## Переменные окружения

| Переменная | По умолчанию (локально) | В Docker Compose |
|---|---|---|
| `DATABASE_URL` | `postgresql+asyncpg://docsearch:docsearch@localhost:5432/docsearch` | `postgresql+asyncpg://docsearch:docsearch@postgres:5432/docsearch` |
| `ELASTICSEARCH_URL` | `http://localhost:9200` | `http://elasticsearch:9200` |

Внутри Docker-сети контейнеры обращаются друг к другу по имени сервиса
(`postgres`, `elasticsearch`) и **внутреннему** порту контейнера (`5432`,
`9200`), а не по порту, проброшенному наружу на хост. Порт Postgres на
хосте вынесен на `5433` (чтобы не конфликтовать с локально установленным
Postgres) — используйте `localhost:5433`, если подключаетесь с хост-машины
(например, через `psql` или DBeaver).
